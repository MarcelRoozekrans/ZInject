# ZInject 1.0 Release — Design Document

**Date:** 2026-03-12
**Author:** Marcel Roozekrans + Claude

## Goal

Ship ZInject 1.0 by completing three remaining workstreams: rebrand from ZeroInject → ZInject, implement `IServiceProviderIsKeyedService`, and set up NuGet packaging with CI/CD.

## Architecture Summary

The project is a Roslyn `IIncrementalGenerator` that emits compile-time DI registrations. It ships three NuGet packages (`ZInject`, `ZInject.Container`, `ZInject.Generator`) targeting net8.0, net9.0, and net10.0. The generator emits hybrid (MS DI fallback) and standalone provider classes inheriting from base classes in `ZInject.Container`.

## 2. `IServiceProviderIsKeyedService`

### Interface

From `Microsoft.Extensions.DependencyInjection.Abstractions` (≥ 8.0.0):

```csharp
public interface IServiceProviderIsKeyedService
{
    bool IsKeyedService(Type serviceType, object serviceKey);
}
```

### Design

Mirrors the existing `IServiceProviderIsService` + `IsKnownService` pattern exactly.

#### Base Classes (conditional — only when keyed services are registered)

**`ZInjectServiceProviderBase`** (hybrid):
- Add `IServiceProviderIsKeyedService` to interface list
- `IsKeyedService(Type, object)`:
  1. Return `IsKnownKeyedService(serviceType, serviceKey)` if true
  2. Delegate to `(_fallback as IServiceProviderIsKeyedService)?.IsKeyedService(serviceType, serviceKey) == true`
- `IsService()` also returns true for `typeof(IServiceProviderIsKeyedService)`
- `GetService()` returns `this` for `typeof(IServiceProviderIsKeyedService)`
- Abstract: `protected abstract bool IsKnownKeyedService(Type serviceType, object serviceKey);`

**`ZInjectStandaloneProvider`** (standalone):
- Same but no fallback delegation

**`ZInjectScope`** + **`ZInjectStandaloneScope`**:
- Delegate `IsKeyedService` to root (same pattern as `IsService`)
- `GetService()` returns root for `typeof(IServiceProviderIsKeyedService)`

#### Conditional Interface Problem

The base classes are compiled once (not generated). But `IServiceProviderIsKeyedService` should only be on the interface list when keyed services exist. Two options:

**Option A: Always implement the interface.** The base classes always implement `IServiceProviderIsKeyedService`, and `IsKnownKeyedService` always returns false when no keyed services exist. Cost: one extra interface per provider. Benefit: no conditional compilation complexity.

**Option B: Generated shim.** The generator emits a partial class that adds the interface. Cost: more generated code. Benefit: zero overhead when no keyed services.

**Decision: Option A.** `IServiceProviderIsKeyedService` is lightweight (single bool method). Always implementing it is consistent with how `IServiceProviderIsService` is already unconditionally implemented. ASP.NET checks for this interface at runtime — having it always present ensures correct behavior even if keyed services are added later via fallback.

#### Generator Changes

New method `EmitIsKnownKeyedService()`:

```csharp
protected override bool IsKnownKeyedService(global::System.Type serviceType, object serviceKey)
{
    if (serviceKey is string _key)
    {
        if (_key == "primary" && serviceType == typeof(global::MyApp.IMyService)) return true;
        if (_key == "secondary" && serviceType == typeof(global::MyApp.IMyService)) return true;
    }
    return false;
}
```

- Only emitted when `hasKeyedServices == true`
- When no keyed services: emit simple `return false;` body
- Groups by key type (currently only `string` keys are supported via `Key = "..."`)

## 3. NuGet Packaging + CI/CD

### Version Strategy

- **GitVersion** calculates semantic version from git tags at build time
- **Release Please** automates release PRs with CHANGELOG from conventional commits
- **Single version** across all three packages

#### GitVersion Configuration (`GitVersion.yml`)

```yaml
mode: ContinuousDeployment
next-version: 0.9.0
branches:
  main:
    regex: ^main$
    mode: ContinuousDelivery
    tag: ''
  feature:
    regex: ^feature/
    mode: ContinuousDeployment
    tag: alpha
  pull-request:
    regex: ^pull/
    mode: ContinuousDeployment
    tag: pr
```

#### Release Please Configuration

- `release-please-config.json` + `.release-please-manifest.json`
- Monorepo mode with single version for all packages
- Conventional commit parsing

### GitHub Actions Workflows

#### 1. `ci.yml` — Continuous Integration

**Trigger:** Push to `main`, PR to `main`

```
Steps:
1. Checkout
2. Setup .NET 8/9/10
3. dotnet restore
4. dotnet build (all TFMs)
5. dotnet test (net10.0)
6. dotnet pack --no-build (dry-run, verify packable)
```

#### 2. `release-please.yml` — Release Automation

**Trigger:** Push to `main`

```
Steps:
1. google-github-actions/release-please-action
2. Creates/updates release PR with CHANGELOG
3. On merge: creates GitHub Release + git tag (v0.9.0, v1.0.0, etc.)
```

#### 3. `publish.yml` — NuGet Publishing

**Trigger:** GitHub Release created (by Release Please)

```
Steps:
1. Checkout at release tag
2. Install GitVersion
3. GitVersion /output buildserver → sets version env vars
4. dotnet pack -p:Version=$GITVERSION_SEMVER
5. dotnet nuget push *.nupkg --source nuget.org --api-key ${{ secrets.NUGET_API_KEY }}
```

### Directory.Build.props Changes

```xml
<Version>0.9.0</Version>  <!-- Default; overridden by CI -->
```

### Branch Rename

`master` → `main` (done before CI setup, one-time operation).

### NuGet Secret

`NUGET_API_KEY` stored as GitHub repository secret (manual setup by maintainer).

## Testing Strategy

### `IServiceProviderIsKeyedService` Tests

1. **Base class unit tests:** `IsKeyedService` returns true for known keyed services, false for unknown
2. **Scope delegation tests:** Scope delegates `IsKeyedService` to root
3. **Generator tests:** Verify `IsKnownKeyedService` override is emitted with correct key+type checks
4. **Generator tests (no keyed):** Verify `IsKnownKeyedService` returns false when no keyed services
5. **Integration tests:** Full compile-and-resolve with `IServiceProviderIsKeyedService` check

### CI Tests

- Verify workflows parse correctly (act or manual)
- Verify `dotnet pack` produces valid `.nupkg` files
- Verify GitVersion reads tags correctly
